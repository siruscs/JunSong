## 一、整体架构与核心原则

本项目基于微服务架构，使用 **Spring Cloud 2025.1.0 + Spring Cloud Alibaba 2025.1.0.0 + Spring Boot 4.0.3**，所有服务通过 **Docker Compose** 编排，在同一网络中通过容器名互相访问。

### 核心原则：所有配置基于 Nacos 配置中心

- 路由规则、数据源、Redis、安全白名单等配置**全部由 Nacos 配置中心管理**
- 本地源码中仅保留服务名、端口和 Nacos 连接信息（`application.yml` / `bootstrap.yml`）
- **禁止**在本地配置文件中定义路由、数据源等业务配置
- **禁止**使用 Java 硬编码路由（如 `RouteLocator` Bean）

### 配置模块化拆分

公共配置按职责拆分为独立的共享配置，避免配置重复：

| 配置文件 | 职责范围 | 说明 |
|---------|---------|------|
| `application-{env}.yml` | 公共基础配置 | Feign、Actuator、MyBatis、Servlet、Logging，`env` 为 `dev` 或 `prod` |
| `redis-{env}.yml` | Redis 连接配置 | 所有服务共用的 Redis 连接池配置 |
| `datasource-druid-{env}.yml` | MySQL + Druid + 动态数据源 | 所有使用 `junsong-common-datasource` 的服务共用 |
| `junsong-xxx-{env}.yml` | 服务专属配置 | 各服务的业务配置，通过 `SPRING_PROFILES_ACTIVE` 选择 dev/prod |

### 请求链路

```
浏览器 → Nginx(:80) → /prod-api/ → Gateway(:8080) → 各微服务
                                         ↓
                                    Nacos 配置中心(:8848)
                                    (路由规则/白名单/共享配置等)
```

## 三、按需启动与最小启动集

项目支持**按需启动**：不必每次都启动全部 13 个服务，可根据要验证的业务模块只启动相关服务，节省本机资源。

### 3.1 最小启动集（能登录进后台）

要让管理员能正常登录后台首页，**最少需要启动以下 7 个服务**：

| # | 服务 | 作用 |
|---|------|------|
| 1 | junsong-mysql | 提供业务库 + Nacos 配置库 |
| 2 | junsong-redis | 缓存验证码、登录令牌、会话 |
| 3 | junsong-nacos | 注册中心 + 配置中心（所有服务依赖） |
| 4 | junsong-auth | 登录认证、签发令牌 |
| 5 | junsong-modules-system | 用户、部门、菜单、角色（登录后需拉取菜单/权限） |
| 6 | junsong-gateway | 统一入口、鉴权、验证码、路由转发 |
| 7 | junsong-nginx | 前端页面与 `/prod-api/` 反向代理 |

**最小启动命令：**

```bash
cd /Users/sirius/Documents/TRAE/JunSong-Cloud/docker

# 1. 基础设施
docker compose up -d junsong-mysql junsong-redis junsong-nacos
sleep 45   # 等待 Nacos 就绪

# 2. 登录链路最小服务
docker compose up -d junsong-auth junsong-modules-system junsong-gateway

# 3. 前端入口
docker compose up -d junsong-nginx
```

> **登录链路原理**：浏览器 → Nginx → Gateway（生成验证码、校验白名单）→ Auth（校验账密、签发令牌）→ Auth 通过 Feign 调用 System 获取用户/部门/角色/菜单。因此 `system` 是登录的硬依赖，缺少它会导致登录后菜单为空或报错。

### 3.2 按业务模块追加启动

在最小启动集基础上，按需追加对应服务：

| 要验证的功能 | 追加启动的服务 |
|------------|--------------|
| 会员 / 积分 / 秒杀 | `junsong-modules-member` |
| 财务 / 进销存 / 投资分润 | `junsong-modules-finance` |
| 工作流 / 低代码 | `junsong-modules-workflow` |
| 文件 / 图片上传 | `junsong-modules-file` |
| 代码生成 | `junsong-modules-gen` |
| 定时任务 | `junsong-modules-job` |
| 服务健康监控 | `junsong-visual-monitor` |

例如，验证低代码审批：

```bash
docker compose up -d junsong-modules-workflow
```

## 八、Nacos 配置管理

> 所有业务配置（路由、数据源、Redis、安全白名单等）均由 Nacos 配置中心管理，**不使用本地业务配置文件**。

### 修改 Nacos 配置

**方式一：通过 Nacos 控制台（推荐）**

1. 访问 http://localhost:7080/next （Nacos 3.x 新版控制台 NConsole）→ 配置管理 → 配置列表
2. 找到目标配置，点击编辑
3. 修改后发布，支持 `@RefreshScope` 的配置会自动热更新

**方式二：从 SQL 备份恢复**

```bash
cd /Users/sirius/Documents/TRAE/JunSong-Cloud
docker exec -i junsong-mysql sh -lc 'mysql --default-character-set=utf8mb4 -uroot -p$MYSQL_ROOT_PASSWORD' \
  < sql/junsong-config-init-20260614.sql
docker restart junsong-nacos
sleep 45
# 然后重启相关业务服务
```

> **注意**：通过 MySQL 直接修改 Nacos 配置后，Nacos 内存缓存不会自动同步，**必须重启 Nacos 服务**。建议优先使用 Nacos 控制台修改。

### 新增路由规则

在 `junsong-gateway-${env}.yml` 中添加：

```yaml
spring:
  cloud:
    gateway:
      server:
        webflux:
          routes:
            - id: 新服务名
              uri: lb://新服务名
              predicates:
                - Path=/新路径/**
              filters:
                - StripPrefix=1
```

> **关键**：Spring Cloud Gateway 5.0.0（Spring Cloud 2025.x）路由前缀已变更为 `spring.cloud.gateway.server.webflux.routes`，旧前缀 `spring.cloud.gateway.routes` 不再生效。

### 新增鉴权白名单

在 `junsong-gateway-${env}.yml` 中添加白名单路径（跳过 Gateway 的 `AuthFilter` 鉴权）：

```yaml
security:
  ignore:
    whites:
      - /auth/login
      - /auth/logout
      - /新路径
```

## 十一、配置加载机制

### Spring Cloud 2025.x 配置加载方式

本项目使用 `spring.config.import` 机制（替代旧版 `bootstrap.yml` 自动加载）。

**Gateway 的 `application.yml`**（本地仅保留连接信息）：

```yaml
server:
  port: 8080
spring:
  application:
    name: junsong-gateway
  profiles:
    active: dev
  config:
    import:
      - optional:nacos:application-${SPRING_PROFILES_ACTIVE}.yml?group=DEFAULT_GROUP&refresh=true
      - optional:nacos:redis-${SPRING_PROFILES_ACTIVE}.yml?group=DEFAULT_GROUP&refresh=true
      - optional:nacos:junsong-gateway-${SPRING_PROFILES_ACTIVE}.yml?group=DEFAULT_GROUP&refresh=true
  cloud:
    nacos:
      discovery:
        server-addr: junsong-nacos:8848
        username: nacos
        password: nacos
      config:
        server-addr: junsong-nacos:8848
        username: nacos
        password: nacos
```

### 配置优先级

```
Nacos 服务专属配置 (junsong-xxx-${env}.yml)  >  Nacos 共享配置 (redis / datasource-druid)  >  Nacos 公共配置 (application-${env}.yml)  >  本地 application.yml
```

### 容器内地址规则

| 配置项 | 容器内值 | 宿主机值 |
|--------|---------|---------|
| Nacos | `junsong-nacos:8848` | `localhost:8848` |
| MySQL | `junsong-mysql:3306` | `localhost:3306` |
| Redis | `junsong-redis:6379` | `localhost:6379` |
| Gateway | `junsong-gateway:8080` | `localhost:8081` |

> Nacos 中的配置使用容器内地址，因为这些配置在容器内生效。

## 十二、常见问题

### 1. 登录页报"令牌不能为空"
**原因**：Gateway 的 `AuthFilter` 拦截了 `/auth/login`，该路径未在白名单。
**解决**：确认 `junsong-gateway-${env}.yml` 的 `security.ignore.whites` 包含 `/auth/login`、`/auth/logout`、`/system/user/deptsForLogin`。

### 2. 登录页报 404 "No static resource system/user/deptsForLogin"
**原因**：Gateway 路由未正确配置或请求未转发到后端。
**解决**：确认路由使用新前缀 `spring.cloud.gateway.server.webflux.routes`，并包含 `/system/**` 规则，重启 Nacos 和 Gateway。

### 3. 验证码不显示
**解决**：确认 `junsong-gateway-${env}.yml` 包含 `security.captcha.enabled: true`，`/code` 在白名单中，重启 Nacos 和 Gateway。

### 4. Nacos 配置修改后不生效
**原因**：通过 MySQL 直接改了 Nacos 配置，内存缓存未同步。
**解决**：`docker compose restart junsong-nacos`，再重启相关业务服务。

### 5. 服务启动报 Client not connected（Nacos gRPC）
**原因**：Nacos 尚未完全就绪。
**解决**：确认 `curl http://localhost:8848/nacos/` 返回 200 后，重启失败服务。

### 6. Docker 构建失败（镜像源 403）
**解决**：`DOCKER_BUILDKIT=0 docker compose build`。

### 7. Gateway 启动后路由不生效
**解决**：确认使用新前缀 `spring.cloud.gateway.server.webflux.routes`（旧前缀 `spring.cloud.gateway.routes` 不再生效）。

### 8. Gen/Job 服务启动报数据源错误
**原因**：Gen/Job 不依赖 `junsong-common-datasource`。
**解决**：Gen/Job **不要**导入 `datasource-druid-${env}.yml`，使用服务专属配置中的 HikariCP 数据源。

### 9. 重建镜像后容器内仍是旧 JAR（Docker 匿名卷问题）
**症状**：`docker compose build` 并 `docker restart` 后，容器内仍是旧 JAR，新接口报 `No static resource`。
**根因**：早期模块 `dockerfile` 的 `VOLUME /home/junsong` 把 JAR 目录变成匿名卷，重建时复用旧卷覆盖新 JAR。
**解决**：删除容器时连带删除匿名卷再重建：
```bash
docker rm -fv junsong-modules-workflow
docker compose up -d junsong-modules-workflow
# 验证容器内 JAR 为最新
docker exec junsong-modules-workflow ls -la /home/junsong/
```
> 长期已移除业务模块 dockerfile 中的 `VOLUME` 指令；`workflow` 模块仍继承 `job` 基础镜像历史层的 VOLUME 元数据，待镜像源恢复后建议改回 `FROM eclipse-temurin:17` 并重建基础镜像。

### 10. workflow/system 模块控制器报 No static resource
**原因**：`workflow`、`system` 模块需显式声明 `spring-boot-starter-web` 与 `junsong-common-core`，否则控制器无法注册到 Spring MVC。
**解决**：确认对应模块 POM 已声明上述依赖（其他模块通过 `junsong-common-swagger` 传递的 webmvc 已具备该能力）。
